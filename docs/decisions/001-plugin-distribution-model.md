# ADR-001: Plugin Distribution Model

Date: 2026-02-28
Status: Accepted

## Context

「手動版RLAnything」のボトルネックを解消する必要があった。開発中の教訓はすべて人間が言語化して MEMORY.md/skills/rules に反映しており、このステップが律速になっていた。スキルの遺伝的最適化と自律進化ループの2層を導入し、教訓の抽出から反映を半自動化することが求められた。

## Decision

- **配布形態は Claude Code Plugin**: 独立リポジトリに分離し、他プロジェクトで再利用可能にする。各プロジェクトへの直接配置や hooks としての実装ではなく、`/optimize` で呼び出せる Skill として配布する
- **評価モデルは3軸統合スコア**: 技術品質 (40%) + ドメイン品質 (40%) + 構造品質 (20%) の3軸。ドメイン品質は CLAUDE.md からドメインを自動推定して評価軸を切替
- **LLM 利用は `claude` CLI 経由**: Anthropic API を直接呼び出すのではなく、`claude -p`（headless mode）で突然変異・交叉・評価を実行。認証管理が不要で既存環境をそのまま活用
- **適応度関数を分離**: 汎用フレームワーク（Plugin）とプロジェクト固有の適応度関数（`scripts/rl/fitness/`）を分離し、Plugin の再利用性を確保

## Alternatives Considered

- **各プロジェクトに直接配置**: 再利用性が低く、プロジェクトごとに個別管理が必要になる
- **単一スコアでの評価**: 技術品質とドメイン品質は独立した軸であり、単一スコアでは適切に表現できない
- **Anthropic API 直接呼び出し**: API キー管理が必要になり、既存の Claude Code 認証環境を活用できない
- **適応度関数を Plugin 内に含める**: プロジェクト固有の評価基準を含めると Plugin の汎用性が失われる

リスク軽減策として、API コストは初期パラメータ最小限（集団3x世代3）で抑制し、モデル崩壊は元スキルを baseline として保持して全劣化時は不採用とし、評価精度は人間確認ステップを必須にして完全自動化を避ける方針とした。

## Consequences

**良い影響:**
- 複数プロジェクトで `/optimize`、`/rl-loop` として即座に利用可能になった
- プロジェクト固有の適応度関数を自由に定義できる拡張性を確保
- 人間確認ステップにより安全な進化サイクルを実現

**悪い影響:**
- Plugin 配布のため、Claude Code Plugin エコシステムへの依存が発生
- 初期は GA ベースの最適化のため LLM コストが高い（後に ADR-003 で解消）
