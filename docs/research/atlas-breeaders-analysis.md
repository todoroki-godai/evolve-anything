## 導入アーキテクチャ: 4層パイプライン

パターンC（ログ解析）+ D（遺伝的最適化）+ E（3役分担）を段階的に積み上げ、claude-reflectをリアルタイム層として組み合わせる。

```
┌─────────────────────────────────────────────────────────┐
│                                                          │
│   C: 観測 ──────► D: 進化 ──────► E: 統合ループ         │
│   (データ収集)    (プロンプト改善)  (自律的に回る)        │
│                                                          │
│   Phase 1         Phase 2          Phase 3               │
│   「何が起きた？」 「何が効く？」   「自分で回せ」         │
│                                                          │
│           ┌───────────────────────┐                      │
│           │ claude-reflect が     │                      │
│           │ リアルタイム層を担当  │                      │
│           └───────────────────────┘                      │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### RLAnythingの3要素との対応

| RLAnything | パターン | このプロジェクトでの役割 |
|---|---|---|
| 環境 (Environment) | C (ログ解析) | 観測データを集める。「何が起きているか」を知る |
| 報酬モデル (Reward) | D (遺伝的最適化) | プロンプトを進化させる。「何が良いか」を探索する |
| 方策 (Policy) | E (3役分担) | 全体を統合して回す。自己進化ループを構成する |

### 4層の詳細

```
Layer 0: claude-reflect (既成品)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  リアルタイム修正キャプチャ
  UserPromptSubmitフックで毎プロンプトをregex走査
  /reflect でレビュー・適用
  → 「明示的な修正」を拾う

Layer 1: C - セッションログ解析 (自作)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  定期的に過去セッションを横断分析
  入力:
    - ~/.claude/projects/.../sessions/*.jsonl
    - subagents/agent-*.jsonl
    - openspec/changes/archive/*/tasks.md
    - git log
  出力:
    - 繰り返しエラーパターン
    - よく書き直されるファイル/関数
    - 成功したタスクの共通行動パターン
    - verify失敗の傾向分析
  → 「無意識の癖」を発掘する

Layer 2: D - スキル/ルールの遺伝的最適化 (自作)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Layer 1 の分析結果をもとに:
    1. スキルSKILL.mdのバリエーションを生成
    2. 実タスクで使用して成功率を測定
    3. 成績の良いバリエーションを採用
  例:
    narrative-ux-writing スキルの指示文を3パターン作成
    → 各パターンでナラティブテキストを生成
    → narrative-quality-check で採点
    → 最高スコアのパターンを正式採用
  → 「どの指示が最も効くか」を実験的に発見

Layer 3: E - 自律ループ (最終形)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Layer 0-2 を統合して自律的に回す:
    メインエージェント → タスク実行
    採点エージェント   → 結果評価 (Layer 1のデータ使用)
    進化エージェント   → プロンプト改善 (Layer 2の手法使用)

  OpenSpec のサイクルに組み込み:
    実装 → verify → 成功/失敗記録 → パターン分析
    → スキル改善 → 次のタスクでより良い実装
  → 「人間が寝てる間も進化が回る」
```

### claude-reflect と C の補完関係

2つは**時間軸が異なる**:

```
claude-reflect（リアルタイム）
━━━━━━━━━━━━━━━━━━━━━━━
  セッション中に「no, use X」→ 即キャプチャ
  セッション後に /reflect → レビュー・適用

  捕まえるもの: 明示的な修正指示
  見逃すもの:  言語化されない暗黙のパターン

パターンC（事後分析・俯瞰）
━━━━━━━━━━━━━━━━━━━━━━━
  過去N週間のセッションログを横断分析
  「同じファイルを3回以上書き直している」
  「tsc エラーが毎回同じパターンで出る」
  「OpenSpec verify で同じ種類の失敗が繰り返される」

  捕まえるもの: 本人が気づいていない繰り返しパターン
  見逃すもの:  リアルタイムの修正指示
```

2つ合わせると「意識的な修正」と「無意識の癖」の両方をカバーできる。

## D（遺伝的最適化）が向くケース

### 最適化対象

| 対象 | 内容 | フィットネス関数 |
|------|------|-----------------|
| skills/*/SKILL.md | 各スキルの指示文 | そのスキル使用時のタスク成功率 |
| rules/*.md | 各ルール | ルール違反の発生頻度 |
| OpenSpecテンプレート | artifact生成プロンプト | verify通過率、イテレーション回数 |
| CLAUDE.md | 全体の行動指針 | セッション全体の生産性指標 |

### なぜこのプロジェクトと相性が良いか

**フィットネス関数の設計という最も難しい部分が、すでに半分できている**:

- `narrative-quality-check` スキル = テキスト品質の**自動採点関数**
- `openspec verify` = 実装品質の**自動採点関数**
- 23スキルそれぞれに「成功率」を測定するパイプラインを組める

## このプロジェクトの強み（他との差別化）

### 1. OpenSpecワークフローが「報酬シグナル」の構造を提供

多くのClaude Code自己改善プロジェクトは「いつ成功/失敗を判定するか」が曖昧。本プロジェクトのOpenSpecは:

```
proposal → design → spec → tasks → 実装 → verify → archive
                                              ↑
                                        明確な判定ポイント
```

### 2. スキル体系が「環境」として既に充実

23スキルはRLAnythingの「環境」に相当。タスクの前提条件や制約を与える仕組みが成熟。

### 3. 段階的開示のドキュメント構造

docs/strategy/ → openspec/ → skills/ → rules/ の階層構造が、RLAnythingの「カリキュラム」に自然にマッピングされる。

### 4. 既存の自動採点関数

narrative-quality-check と openspec verify が、遺伝的最適化のフィットネス関数として機能する。

## 段階的導入計画

### Phase 1: リアルタイム層（1日）

claude-reflect をインストールし、日常開発に組み込む。

```
claude plugin marketplace add bayramannakov/claude-reflect
claude plugin install claude-reflect@claude-reflect-marketplace
```

- CJK偽陽性の確認・対応
- メモリ階層ルーティングの調整（本プロジェクトの rules/skills/memory 構造に合わせる）
- OpenSpec verify 後の `/reflect` 実行を習慣化

### Phase 2: 観測層 - Layer 1: C（2週間）

セッションログ横断分析のパイプラインを構築。

入力データ:
- `~/.claude/projects/.../sessions/*.jsonl`
- `subagents/agent-*.jsonl`
- `openspec/changes/archive/*/tasks.md`
- `git log`

分析項目:
- 繰り返しエラーパターンの検出
- よく書き直されるファイル/関数の特定
- 成功タスクの共通行動パターン
- verify失敗の傾向分析

出力:
- 週次レポート（`docs/research/reports/`）
- MEMORY.md / スキル改善提案

### Phase 3: 進化層 - Layer 2: D（1ヶ月）

スキル/ルールの遺伝的最適化を導入。

最初のターゲット:
- `narrative-ux-writing` SKILL.md（フィットネス関数: narrative-quality-check のスコア）
- OpenSpec tasks.md テンプレート（フィットネス関数: verify通過率）

手順:
1. SKILL.md のバリエーションを3つ生成
2. 各バリエーションで同じタスクを実行
3. フィットネス関数で採点
4. 最高スコアのバリエーションを採用
5. 次世代のバリエーションを生成（遺伝的操作: 交叉 + 変異）

### Phase 4: 統合ループ - Layer 3: E（2ヶ月）

Layer 0-2 を統合した自律進化ループを構築。

```
OpenSpec サイクル
  実装 → verify → 成功/失敗記録
    → Layer 1 が分析
    → Layer 2 がスキル改善
    → 改善されたスキルで次のタスク
```

3つのエージェント役割:
- **メインエージェント**: タスク実行（既存のClaude Code）
- **採点エージェント**: 結果評価（サブエージェント / OpenSpec verify拡張）
- **進化エージェント**: プロンプト改善（Layer 2の自動化）

## リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| MEMORY.md 200行制限の超過 | コンテキスト品質低下 | 定期枝刈り。古い教訓は別ファイルに退避 |
| 23スキルの整合性維持 | スキル間の矛盾 | `/reflect --dedupe` で定期チェック |
| 自動追記によるノイズ | 不正確な教訓の蓄積 | 人間レビュー維持（`/reflect` 方式） |
| フック実行のパフォーマンス | 開発体験の悪化 | regexベースで軽量（ミリ秒単位） |
| CJK偽陽性 | 日本語で誤検出 | PR #19対応を確認、必要なら自前フィルタ |
| 遺伝的最適化のコスト | API使用量増加 | 最初は手動で月1回、段階的に自動化 |
| モデル崩壊 | 自己生成データで品質退化 | 人間レビューをループに保持 |
