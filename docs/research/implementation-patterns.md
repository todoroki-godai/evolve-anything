# 実装パターンと落とし穴

## 5つの実装パターン

### パターンA: リフレクション・ループ（最もシンプル）

```
ミス発生
  → ユーザーが「Reflect on this mistake」と指示
  → Claude がパターンを抽象化
  → CLAUDE.md に自動追記
  → 次のセッションで反映
```

- **参考実装**: [aviadr1/claude-meta](https://github.com/aviadr1/claude-meta)
- **コスト**: 極小（CLAUDE.mdにメタルール1行追加するだけ）
- **効果**: 中〜高
- **向いている場面**: 初めて自己進化を試すとき

### パターンB: フック駆動型自動学習

```
PostToolUseフック → 行動結果を検証/採点
  → Stopフック → セッション終了時にリフレクション実行
  → 学習をスキルファイルまたはCLAUDE.mdに書き戻し
```

- **参考実装**: [haddock-development/claude-reflect-system](https://github.com/haddock-development/claude-reflect-system)、[SpecWeave Reflect](https://spec-weave.com/docs/skills/extensible/self-improving-skills/)
- **コスト**: 中（フック設定 + リフレクションスクリプト）
- **効果**: 高
- **向いている場面**: 日常的な開発フローに組み込みたいとき

### パターンC: セッションログ解析型

```
過去のJSONLセッションログを定期的にスキャン
  → フラストレーションパターン・繰り返し要求を特定
  → CLAUDE.md の改善提案を生成
```

- **参考実装**: [martinalderson.com の手法](https://martinalderson.com/posts/self-improving-claude-md-files/)
- **コスト**: 低（分析スクリプト作成）
- **効果**: 中
- **向いている場面**: 既に蓄積されたログがあるとき（本プロジェクトに該当）

### パターンD: 遺伝的アルゴリズムによるプロンプト最適化

```
初期プロンプト集団を生成
  → LLMガイドの遺伝的操作で変異
  → 評価関数でフィットネス測定
  → 次世代の選択
  → 反復して最適プロンプトに収束
```

- **参考**: [GAAPO](https://www.frontiersin.org/journals/artificial-intelligence/articles/10.3389/frai.2025.1613007/full)
- **コスト**: 高（評価関数の設計、計算リソース）
- **効果**: 最高（ただし設計が難しい）
- **向いている場面**: 特定タスクのプロンプトを徹底的に最適化したいとき

### パターンE: 3役分担型（RLAnything直接適用）

```
メインエージェント（方策）→ タスクを実行
  ↕
サブエージェント（報酬モデル）→ 実行結果を採点
  ↕
環境マネージャー → タスク難易度を調整、コンテキストを更新
  ↓
3つのフィードバックループで同時に進化
```

- **参考**: [Gen-Verse/Open-AgentRL](https://github.com/Gen-Verse/Open-AgentRL)
- **コスト**: 最高（サブエージェント構成、評価基準設計、難易度調整ロジック）
- **効果**: 最高
- **向いている場面**: フルスケールで自己進化を実現したいとき

## パターン間の関係性

### C + D + E = RLAnythingの3要素

5つのパターンはバラバラではなく、C・D・Eは**RLAnythingの3要素そのもの**に対応する:

| RLAnything | パターン | 役割 |
|---|---|---|
| 環境 (Environment) | **C** (ログ解析) | 観測データを集める。「何が起きているか」を知る |
| 報酬モデル (Reward) | **D** (遺伝的最適化) | プロンプトを進化させる。「何が良いか」を探索する |
| 方策 (Policy) | **E** (3役分担) | 全体を統合して回す。自己進化ループを構成する |

```
  C: 観測 ──────► D: 進化 ──────► E: 統合ループ
  (データ収集)    (プロンプト改善)  (自律的に回る)

  Phase 1         Phase 2          Phase 3
  「何が起きた？」 「何が効く？」   「自分で回せ」

          ┌───────────────────────┐
          │ A+B (claude-reflect)  │
          │ リアルタイム層を担当  │
          └───────────────────────┘
```

### パターン選択ガイド

```
「今日から始めたい」
  → パターンA+B（claude-reflect）

「蓄積されたログを活用したい」
  → パターンC（ログ解析）← A+Bと補完関係

「特定スキルを極限まで最適化したい」
  → パターンD（遺伝的最適化）← Cのデータを使う

「本格的にRLAnythingを再現したい」
  → パターンE（3役分担）← C+Dを統合して自律化
```

推奨の進め方: **A+B → C → D → E の段階的積み上げ**

詳細な導入計画: [atlas-breeaders-analysis.md](./atlas-breeaders-analysis.md)

## 落とし穴と対策

### 1. ルール肥大化（最も頻出）

| 症状 | CLAUDE.md / Memory が際限なく膨張し、コンテキスト品質が低下 |
|------|-------|
| 原因 | 自動追記にブレーキがない |
| 対策 | 定期的な重複排除・枝刈り。信頼度スコアに基づく取捨選択。claude-reflectの`--dedupe`が参考になる |
| 本プロジェクトでの懸念 | MEMORY.mdは200行制限あり（CLAUDE.md指示）。スキル23個はすでに多い |

### 2. モデル崩壊（Model Collapse）

| 症状 | 再帰的に自己生成データで学習→品質が退化する |
|------|-------|
| 原因 | AIが自分の出力を正解として学習し続ける再帰ループ |
| 対策 | 人間レビューをループに保持。自動追記は「提案」に留め、人間が承認 |
| 本プロジェクトでの対策案 | `/reflect` コマンド方式で人間の確認ステップを保持 |

### 3. 過学習（Overfitting to Specific Errors）

| 症状 | 特定のエラーパターンに過度に特化し、一般性を失う |
|------|-------|
| 原因 | 1回の失敗から過度に一般化したルールを作成 |
| 対策 | パターンの抽象化レベルを意識。複数回確認されたパターンのみルール化 |
| 本プロジェクトの既存対策 | rules-style.md「3行以内で書く」がすでに抽象化を強制している |

### 4. ガードレール不在

| 症状 | 自動最適化がコーディング規約やセキュリティ基準を逸脱 |
|------|-------|
| 原因 | 自動生成ルールが既存ルールと矛盾 |
| 対策 | PostToolUseフックでバリデーション。「不変のルール」と「進化するルール」を分離 |
| 本プロジェクトでの分離案 | `.claude/rules/`（不変）vs `memory/`（進化可能） |

### 5. 観測コストの爆発

| 症状 | 全行動をログすると、ログ量が膨大になり分析が困難 |
|------|-------|
| 原因 | PostToolUseフックで無差別に全ログ取得 |
| 対策 | フィルタリング（エラー時のみ詳細ログ等）。要約ベースの記録 |
| 参考 | hooks-masteryのBuilder/Validatorパターンは選択的ログ |
