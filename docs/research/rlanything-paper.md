## 従来のRLの問題

### 1. 報酬の疎さ（Reward Sparsity）

コーディングのような長いタスクでは、最終結果（テスト通過/失敗）でしか報酬が得られない。途中の良い判断が評価されず、学習効率が悪い。

### 2. 教師のボトルネック（Teacher Bottleneck）

報酬を設計する人間がAIの成長スピードに追いつけない。AIが1秒で100個の行動を出力しても、人間がチェックするのは不可能。

## RLAnythingの核心：3要素の同時最適化

従来のRLは**方策（Policy）だけ**を学習し、報酬と環境は人間が固定していた。RLAnythingは3つを同時に最適化する。

```
┌──────────────────────────────────────────────────────┐
│                                                       │
│  ┌──────────┐    ┌──────────────┐    ┌──────────┐   │
│  │  方策    │    │  報酬モデル  │    │  環境    │   │
│  │(Policy)  │◄──►│(Reward Model)│◄──►│(Environ) │   │
│  │行動を決定│    │採点基準を改善│    │課題を調整│   │
│  └──────────┘    └──────────────┘    └──────────┘   │
│       │               │                │             │
│  3つのフィードバックループで相互に進化                │
│                                                       │
│  ① 統合FB:   途中経過 + 最終結果 → 方策改善          │
│  ② 一貫性FB: 結果から逆算 → 採点基準を修正           │
│  ③ 評価FB:   成功率に応じて → 課題難易度を自動調整   │
│                                                       │
└──────────────────────────────────────────────────────┘
```

### 統合フィードバック

途中経過の評価と最終結果を合わせて方策を改善する。「テストは落ちたけど設計は良かった」なら設計部分は正のフィードバック。全否定ではなく、良かった点を評価。

### 一貫性フィードバック

最終的に成功した行動を見て、途中の判断の良し悪しを逆算で学習。例: 「行数が短いコードが良い」という初期の採点基準 → 実際は「コメントがちゃんと書いてあるコードの成功率が高い」と学習して修正。

### 評価フィードバック

成功率約80%が最も学習効率が良い。成功率が高すぎたら問題を難しくし、低すぎたら具体的な指示に変える。ゲームの動的難易度調整と同じ原理。

## ベンチマーク成果

| ベンチマーク | モデル | 改善幅 |
|-------------|--------|--------|
| OSWorld（PC操作） | Qwen3-VL-8B-Thinking | **+9.1%** |
| AlfWorld（テキスト環境操作） | Qwen2.5-7B-Instruct | **+18.7%** |
| LiveBench（汎用推論） | Qwen2.5-7B-Instruct | **+11.9%** |

特筆すべきは、**AIが自分で作った採点基準が人間のラベルより精度が高かった**こと（人間約80% → AI最適化後約85%）。
