## Context

rl-anything は `/optimize` で遺伝的アルゴリズムによるスキル最適化を提供し、`/audit` で環境の健康診断レポートを出力する。`/optimize` 実行時のベースラインスコアは `generations/{run_id}/result.json` に保存され、observe hooks は `usage.jsonl` にスキル使用回数を記録する。

しかし現状では、高頻度で使われる global スキルの品質が時間の経過やスキル本体の編集により劣化しても、それを検知する仕組みがない。ユーザーは `/optimize` を手動で実行して初めてスコアを確認できるが、どのスキルを優先的に最適化すべきかの判断材料がない。

既存コンポーネント:
- `optimize.py` の `_llm_evaluate()` — CoT 付き品質評価（clarity/completeness/structure/practicality の4軸）
- `audit.py` — 環境棚卸し + 使用状況集計 + Scope Advisory レポート
- `observe.py` — PostToolUse hook で usage.jsonl / usage-registry.jsonl に記録
- `~/.claude/skills/` — global スキルの配置先

## Goals / Non-Goals

**Goals:**
- 高頻度 global スキルの品質スコアを定期的に計測し、履歴として追跡する
- `/audit` レポートで品質推移とスコア劣化を視覚的に確認できるようにする
- 一定使用回数（50回）または一定期間（7日）で自動的に再スコアリングをトリガーする
- 劣化検知時（10%以上のスコア低下）に `/optimize` 推奨を通知する
- 既存の `_llm_evaluate()` ロジックを再利用し、評価基準の一貫性を保つ

**Non-Goals:**
- 自動的に `/optimize` を実行すること（通知のみ、実行は人間判断）
- project-scope スキルの品質モニタリング（将来検討）
- リアルタイム監視（バッチ実行のみ）
- 評価関数自体の品質モニタリング（evolve-fitness の責務）

## Decisions

### D1: 品質スコア計測 — quality_monitor.py が独自に `claude -p` を呼び出す

**選択**: `quality_monitor.py` が `claude -p` を直接呼び出し、optimize.py の `_llm_evaluate()` と同一のプロンプトで品質評価を実行する。optimize.py のリファクタリングは行わない。

**代替案A**: `optimize.py` の `_llm_evaluate()` を `scripts/lib/` に抽出して共通化する → optimize.py には 73 テストが安定稼働中であり、リファクタリングは breaking change リスクがある。このchangeのスコープ外とする
**代替案B**: `optimize.py` を直接 import する → GeneticOptimizer のインスタンス化が必要になり、不要な依存が発生

**根拠**: optimize.py への変更リスクを回避しつつ、同一プロンプトを使用することで評価基準の一貫性を確保する。将来的に共通化する際は別changeとして扱う。

### D2: ベースライン保存先 — ~/.claude/rl-anything/quality-baselines.jsonl

**選択**: 既存の DATA_DIR（~/.claude/rl-anything/）に JSONL 形式で追記。各レコードにスキル名・スコア・タイムスタンプ・使用回数を含む。

**代替案**: generations/ 配下に保存 → optimize 固有のデータと混在し管理が複雑化

**根拠**: usage.jsonl/errors.jsonl と同じパターンに従うことで、audit.py からの読み込みが自然に統合できる。

### D3: 再スコアリングトリガー — 使用回数 50 回 OR 7日経過

**選択**: 前回計測からの使用回数が 50 回以上、または前回計測から 7 日以上経過した場合に再スコアリングをトリガーする。どちらか一方を満たせばトリガー。

**代替案**: 使用回数のみ → 低頻度スキルの劣化を見逃す
**代替案**: 期間のみ → 高頻度スキルで遅延する

**根拠**: 高頻度スキルは使用回数で素早く検知し、低頻度スキルは期間でカバーする二重トリガー方式。

### D4: 劣化判定閾値 — ベースラインから 10% 以上のスコア低下

**選択**: 直近のスコアが最高スコア（ベースライン）から 10% 以上低下した場合に劣化と判定する。

**代替案**: 固定閾値（例: 0.7 未満） → スキルごとの基準が異なるため不適切
**代替案**: 直前スコアとの比較のみ → 緩やかな劣化を見逃す

**根拠**: 相対比較により、スキルごとの品質水準に適応した劣化検知が可能。累積的な劣化も検知できる。

### D5: 高頻度スキルの判定 — usage.jsonl で直近30日に 10回以上使用

**選択**: `audit.py` の `load_usage_data()` + `aggregate_usage()` を利用し、直近30日で10回以上使用された global スキルを「高頻度」と判定する。

**代替案**: 全 global スキルを対象 → LLM コスト増大（未使用スキルまで評価）

**根拠**: 実際に使われているスキルのみを評価対象とすることで、コストを抑えつつ実用的な監視を実現。

### D6: audit レポートへの統合 — 品質推移セクションを generate_report() に追加

**選択**: `audit.py` の `generate_report()` に品質推移セクションを追加。スコア履歴をスパークライン風（Unicode ブロック文字）で視覚化し、劣化警告と optimize 推奨を表示する。

**代替案**: 別コマンドとして実装 → ユーザーが複数コマンドを実行する必要あり

**根拠**: `/audit` は既に環境の健康診断を担っており、品質推移はその自然な拡張。1画面レポートの方針に合致。

### D7: /audit 実行時の品質計測統合フロー

**選択**: `/audit` 実行時に quality_monitor.py の再スコアリング判定を呼び出し、必要なスキルがあれば品質計測を実行してからレポートを生成する。`--skip-rescore` オプションで計測をスキップし、既存のキャッシュ済みスコアのみでレポートを生成できる。

**代替案**: audit と品質計測を完全に分離する → ユーザーが2つのコマンドを手動で実行する必要あり

**根拠**: `/audit` は環境の健康診断を一括で行うコマンドであり、品質計測もその一部として自然に統合できる。計測が不要な場合は `--skip-rescore` でスキップ可能にすることで、実行時間とLLMコストを制御できる。

## Risks / Trade-offs

| リスク | 緩和策 |
|--------|--------|
| LLM 評価コスト（品質計測時） | 高頻度スキルのみ対象（10回以上/30日）+ 再スコアリング間隔制限（50回 or 7日） |
| LLM 評価のばらつき（同一スキルでもスコアが変動） | 直近3回の移動平均を使用し、一時的な変動をスムーズ化 |
| audit 実行時間の増加 | `--skip-rescore` で品質計測をスキップ可能。デフォルトでは再スコアリング判定→必要時のみ計測 |
| ベースラインが存在しないスキル | 初回は計測のみ（劣化判定なし）。2回目以降から比較開始 |
| global スキルの特定精度 | audit.py の既存 classify_artifact_origin() を再利用 |
| LLM 評価のタイムアウト | 計測をスキップし既存レコード不変。エラーを stderr に出力 |
