## Context

現在の mutation プロンプトは「改善方針をランダムに1-2個選んで適用」と指示するだけで、変異の強度やスコープを制御できない。LLM は「改善 = 簡潔化」と解釈する傾向があり、文字数が大幅に減少して情報が欠落する。crossover も同様の傾向がある。

現在の mutation プロンプト:
```
改善方針（ランダムに1-2個選んで適用）:
- より具体的な例を追加
- 曖昧な指示を明確化
- 構造を整理
- 不要な冗長性を削除
- エッジケースの対処を追加
```

## Goals / Non-Goals

**Goals:**
- mutation で情報を保持しつつ質を上げる
- mutation 強度を3段階で制御可能にする
- 高品質スキルに対する無駄な最適化を防ぐ

**Non-Goals:**
- 完全に新しい遺伝的アルゴリズムの導入
- fitness 関数自体の変更

## Decisions

### 1. mutation プロンプトの改善

情報保持を明示的に制約として追加:

```
**重要な制約**:
- 元のスキルの情報量を維持または増加させること。情報の削除は禁止
- 元のスキルの行数の ±20% 以内に収めること
- 各セクションの目的と内容を保持すること
```

### 2. mutation 強度パラメータ

`--mutation-strength` CLI 引数で3段階を選択:

| 強度 | 変異スコープ | プロンプト指示 |
|------|-------------|---------------|
| `light` | 表現改善・タイポ修正・語順調整 | 「内容を変えず、表現のみ改善」 |
| `medium` (default) | セクション内の加筆・整理 | 現行の改善方針（情報保持制約付き） |
| `heavy` | セクション追加/統合・構造再設計 | 「構造を大胆に再設計してよい」 |

`GeneticOptimizer.__init__()` に `mutation_strength` パラメータを追加。

### 3. 部分的 mutation

セクション（`##` 見出し）単位でスキルを分割し、ランダムに1-2セクションのみ mutation 対象として選択。残りのセクションはそのまま保持。

**実装**: `mutate()` 内で `## ` で split → ランダム選択 → 選択セクションのみ LLM に渡す → 結果を元のセクション群に再結合。

**代替案**: 全体を渡して「このセクションだけ変更して」と指示 → 却下: LLM は指示を無視して全体を変更する傾向が強い。

### 4. 高品質スキル検出

ベースライン（Gen 0 の elite）スコアが `HIGH_QUALITY_THRESHOLD = 0.85` 以上の場合:
- 警告メッセージ「ベースラインスコアが高いため、改善余地が限られます」を表示
- `mutation_strength` を自動的に `light` に切り替え
- `--force` フラグで上書き可能

## Risks / Trade-offs

- **行数制約の厳格化**: ±20% 制約により、本来有益な大幅追加が抑制される可能性 → `heavy` モードでは制約を緩和（±50%）
- **部分的 mutation の品質**: セクション単位の分割・再結合で文脈が断裂する可能性 → セクション間の参照関係がある場合は隣接セクションも文脈として渡す
- **高品質検出の閾値**: 0.85 が適切かは運用データで調整が必要 → 設定ファイルで変更可能にする
