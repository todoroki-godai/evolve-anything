## Context

`skills/audit/scripts/audit.py`（366行目）の `_extract_paths_outside_codeblocks()` は、MEMORY/CLAUDE.md ファイル内のファイルパスを正規表現で検出する。正規表現 `r'(?:^|[\s`"\''])(/[a-zA-Z0-9_./-]{2,}|[a-zA-Z0-9_.-]+/[a-zA-Z0-9_./-]+)'` は `word/word` パターンすべてにマッチするため、過度に広い。正規表現マッチの後に3つのポストフィルタが存在する:

1. 長さ < 3 または `http` プレフィックス除外
2. スラッシュコマンド除外（2番目の `/` がない絶対パス）
3. 全大文字セグメント除外（Python シンボル参照）

これらのフィルタは、MEMORY でよく見られる説明的略記パターン `usage/errors`（「usage と errors のレコード」の意味）や `discover/audit`（「discover と audit の機能」の意味）を見逃す。これらの偽陽性は `collect_issues()` を通じて `stale_ref` issue として流れ、Remediation エンジンで自動修正削除がトリガーされる可能性がある。

この関数を利用する2箇所:
- `audit.py` 541行目（レガシー `build_report`）
- `audit.py` 1077行目（`collect_issues`）

どちらも抽出パスに対して `Path.exists()` をチェックする。偽陽性は常にこのチェックに失敗し（実際のパスではないため存在しない）、常に stale_ref issue になる。

## Goals / Non-Goals

**Goals:**
- `_extract_paths_outside_codeblocks()` の MEMORY 説明パターンに対する偽陽性を削減
- 既存の真陽性検出をすべて維持（拡張子付きパス、深いパスなど）
- 抽出関数の専用ユニットテストカバレッジを追加

**Non-Goals:**
- 偽陽性をゼロにすること（一部のエッジケースは残る。ヒューリスティックは本質的に不正確）
- 正規表現自体の変更（正規表現は候補の捕捉として正しい。フィルタリングが適切なレイヤー）
- 下流の利用者の変更（remediation.py, collect_issues）

## Decisions

### Decision 1: 正規表現変更ではなくポスト正規表現ヒューリスティックフィルタ

**選択**: 既存の正規表現マッチの後、結果リストへの追加前にフィルタを追加する。

**根拠**: 正規表現は広い候補セットを正しく捕捉している。正規表現を狭めると、実際のパスを見逃すリスクがある。ポストフィルタはテスト・拡張・説明がより容易。

**検討した代替案**: ファイル拡張子を要求するよう正規表現を書き直す — `skills/update/`（拡張子なしのディレクトリ）のような有効な参照を見逃すため却下。

### Decision 2: 既知プレフィックスによる2セグメントフィルタ（単一戦略）

**選択**: セグメントが正確に2つでどちらにもファイル拡張子がないパスの場合、最初のセグメントが既知のプロジェクトディレクトリパターン（例: `skills/`, `scripts/`, `hooks/`, `.claude/`）にマッチすることを要求する。これにより `usage/errors`, `discover/audit` を排除しつつ `skills/update`, `scripts/lib` を保持する。

**根拠**: 既知プレフィックスフィルタ単独で、全既知偽陽性（`usage/errors`, `discover/audit`, `observe/hooks`）をカバーできる。複数段階の戦略は不要な複雑性を持ち込む。

**検討した代替案**:
- Markdown リスト項目コンテキストフィルタ — 既知プレフィックスでないパスは本フィルタで除外済み。既知プレフィックスの場合（例: `skills/update`）は、リスト項目内でもパス参照として正しいため不要。
- 既知の非パスワードペアの deny-list — 既知プレフィックスフィルタで汎化済み。手動メンテナンスコストが増えるだけで効果なし。
- LLM ベースの分類 — パターンマッチング問題には過剰で、レイテンシ/コストが増加するため却下。

### Decision 3: 設定可能な既知ディレクトリプレフィックス

**選択**: 既知のプロジェクトディレクトリプレフィックス（`KNOWN_DIR_PREFIXES`）のセットを `audit.py` のモジュールレベル定数として定義する。これらのプレフィックスで始まる2セグメントパスは実際のパスである可能性が高いとして扱う。

**根拠**: rl-anything プロジェクトは一貫したディレクトリ構造を持つ。大多数のケースを正しく分類する最もシンプルなヒューリスティック。

### Decision 4: 専用テストファイル

**選択**: パラメータ化テスト付きの `skills/audit/scripts/tests/test_path_extraction.py` を作成する。

**根拠**: この関数には現在専用テストがない（`test_collect_issues.py` によるインテグレーションレベルのカバレッジのみ）。パラメータ化テストにより、新しい偽陽性ケースの追加が発見時に容易になる。

## Risks / Trade-offs

- **[偽陰性]** 既知プレフィックスアプローチは、通常と異なるディレクトリ構造の実際のパスを見逃す可能性がある。緩和策: プレフィックスリストは容易に拡張可能で、標準的なプロジェクトレイアウトをカバーしている。
- **[2セグメント偏重]** フィルタは2セグメントパスに焦点を当てている。より深い偽陽性（例: `a/b/c` がすべて説明的）は対処されない。緩和策: 実際にはまれ。MEMORY ファイル内の3セグメント以上のパスはほぼすべて実際のパス。
