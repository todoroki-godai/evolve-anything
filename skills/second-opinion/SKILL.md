# セカンドオピニオン

Claude Agent（`second-opinion` エージェント）を使って、問題定義や設計案に対する独立した見解を取得する。
codex CLI が不要で、Claude Code の Agent ツールのみで動作する。

## 実行手順

### Step 1: モード判定

引数またはコンテキストからモードを判定する:

- `startup` — プロダクト・ビジネスアイデアの評価（デフォルト: office-hours から呼ばれた場合）
- `builder` — 個人プロジェクト・ハッカソン・学習目的の評価
- `general` — 設計・アーキテクチャ・技術選定のレビュー

引数にモードが明示されていない場合、会話の文脈から推測する。
判断が難しい場合は AskUserQuestion で確認する:

```
セカンドオピニオンのモードを選んでください:
A) Startup — プロダクト/ビジネスアイデアの評価
B) Builder — 個人PJ/ハッカソン/学習の評価
C) General — 設計/アーキテクチャレビュー
```

### Step 2: コンテキスト収集

セカンドオピニオンの品質はコンテキストの質に依存する。以下を収集する:

**必須:**
- **問題定義**: 何を解決しようとしているか
- **前提**: 合意された前提条件のリスト

**任意（あれば品質向上）:**
- **調査結果**: WebSearch や競合分析の結果
- **コードベース情報**: プロジェクト名、言語、最近の活動
- **設計案**: 現在検討中の設計やアーキテクチャ
- **議論の経緯**: これまでに検討した選択肢とその評価

収集方法:
1. 会話履歴からこれらの情報を抽出する（office-hours 後なら Phase 1-3 の内容）
2. 不足している必須情報があれば AskUserQuestion で補完する
3. ユーザーが引数でコンテキストを直接渡した場合はそれを使う

### Step 3: 構造化コンテキストブロックの組み立て

収集した情報を以下のフォーマットに構造化する。
会話の流れに影響されない cold-read を実現するため、要約形式にする:

```
MODE: {startup|builder|general}

PROBLEM:
{問題定義を 2-3 文で}

PREMISES:
1. {前提 1}
2. {前提 2}
...

FINDINGS:
{調査結果の要約。なければ "なし"}

CODEBASE:
{プロジェクト情報。なければ "なし"}

DESIGN:
{設計案の要約。なければ "なし"}
```

ユーザーの発言を含める場合は verbatim（原文そのまま）で引用する。
要約者（あなた）のバイアスがかからないよう注意する。

### Step 4: エージェント起動

Agent ツールで `second-opinion` エージェントを起動する。
プロンプトには Step 3 で組み立てたコンテキストブロックをそのまま渡す:

```
以下のコンテキストに対して、独立したセカンドオピニオンを提供してください。
あなたはこの会話の履歴を持っていません。渡されたコンテキストのみで判断してください。

{コンテキストブロック}
```

エージェント設定:
- subagent_type は指定しない（agents/second-opinion.md が自動解決される）
- name: "second-opinion"
- model は指定しない（エージェント定義の sonnet を使用）

### Step 5: 結果の提示

エージェントの出力を verbatim で提示する:

```
SECOND OPINION (Claude Agent):
════════════════════════════════════════════════════════════
{エージェント出力をそのまま。要約や編集をしない}
════════════════════════════════════════════════════════════
```

### Step 6: クロスパースペクティブ合成

エージェント出力を提示した後、3-5 箇条書きで合成を提供する:

- **一致点**: 自分（メインセッションの Claude）とセカンドオピニオンが同意する点
- **相違点**: 意見が分かれる点と、その理由
- **前提への影響**: セカンドオピニオンが前提に挑戦した場合、それが推奨にどう影響するか

### Step 7: ユーザーアクション（任意）

セカンドオピニオンが前提に挑戦した場合、ユーザーに確認する:

```
セカンドオピニオンが前提「{前提テキスト}」に挑戦しています。
A) この前提を修正する
B) 元の前提を維持する
```

ユーザーが理由を添えて B を選んだ場合、それは重要なシグナルとして記録する。

## エラーハンドリング

- エージェント起動に失敗した場合: エラーを報告し、手動でのレビューを提案する
- エージェントが空の応答を返した場合: 再試行を 1 回だけ行い、それでもダメなら報告する
- タイムアウト: Agent ツールのデフォルトタイムアウトに従う

## 使用例

```
# office-hours 後にセカンドオピニオン
/rl-anything:second-opinion startup

# 設計レビュー
/rl-anything:second-opinion general

# モード自動判定（会話文脈から）
/rl-anything:second-opinion

# コンテキスト直接指定
/rl-anything:second-opinion general "マイクロサービス vs モノリスの設計判断"
```
