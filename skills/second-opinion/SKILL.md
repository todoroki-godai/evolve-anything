# セカンドオピニオン

問題定義や設計案に対する独立した見解を、2つのルートのいずれかで取得する。

- **ルートA（既定）**: Claude Agent（`second-opinion` エージェント）による cold-read。
  Claude Code の Agent ツールのみで動作し、日常の軽い二次意見に向く。
- **ルートB（新規・条件付き発動）**: codex CLI（GPT 系・別モデル系統）による外部 cold-read。
  既存のルートAは Claude agent が Claude を審査する構造のためエラーが相関しうる。
  codex が使える環境では、真に独立した第三者視点を取得できる。

## 実行手順

### Step 0: ルート判定（codex 検出）

Bash ツールで codex CLI の有無を確認する: `command -v codex`

- 検出できなかった場合 → ルートAのみ。Step 1 へ進む。
- 検出できた場合、以下のいずれかを満たせばルートBを使う:
  - ユーザーが「codexで」「外部の視点で」等、外部ツールの利用を明示している
  - ship 前・大きな設計判断など高リスクな局面で、より独立性の高い意見が必要と判断できる
  - 上記どちらにも該当しない（日常の軽い確認）場合はルートAを使う

判断が難しい場合は AskUserQuestion で確認する:

```
codex CLI を検出しました。セカンドオピニオンのルートを選んでください:
A) ルートA — Claude Agent（速い・日常向け）
B) ルートB — codex 外部 cold-read（別モデル系統・独立性が高い、ship前/大きな設計判断向け）
```

ルートBを選び `codex exec` の実行に失敗した場合（非ゼロ終了・エラー出力）は、
握りつぶさずルートAへフォールバックする。その際は必ず
「codex 不可のためルートA で実行します（理由: {エラー概要}）」とユーザーに表示してから
Step 1 以降をルートAとして続行する。

### Step 1: モード判定

引数またはコンテキストからモードを判定する:

- `startup` — プロダクト・ビジネスアイデアの評価（デフォルト: office-hours から呼ばれた場合）
- `builder` — 個人プロジェクト・ハッカソン・学習目的の評価
- `general` — 設計・アーキテクチャ・技術選定のレビュー

引数にモードが明示されていない場合、会話の文脈から推測する。
判断が難しい場合は AskUserQuestion で確認する:

```
セカンドオピニオンのモードを選んでください:
A) Startup — プロダクト/ビジネスアイデアの評価
B) Builder — 個人PJ/ハッカソン/学習の評価
C) General — 設計/アーキテクチャレビュー
```

### Step 2: コンテキスト収集

セカンドオピニオンの品質はコンテキストの質に依存する。以下を収集する:

**必須:**
- **問題定義**: 何を解決しようとしているか
- **前提**: 合意された前提条件のリスト

**任意（あれば品質向上）:**
- **調査結果**: WebSearch や競合分析の結果
- **コードベース情報**: プロジェクト名、言語、最近の活動
- **設計案**: 現在検討中の設計やアーキテクチャ
- **議論の経緯**: これまでに検討した選択肢とその評価

収集方法:
1. 会話履歴からこれらの情報を抽出する（office-hours 後なら Phase 1-3 の内容）
2. 不足している必須情報があれば AskUserQuestion で補完する
3. ユーザーが引数でコンテキストを直接渡した場合はそれを使う

### Step 3: 構造化コンテキストブロックの組み立て

収集した情報を以下のフォーマットに構造化する。
会話の流れに影響されない cold-read を実現するため、要約形式にする:

```
MODE: {startup|builder|general}

PROBLEM:
{問題定義を 2-3 文で}

PREMISES:
1. {前提 1}
2. {前提 2}
...

FINDINGS:
{調査結果の要約。なければ "なし"}

CODEBASE:
{プロジェクト情報。なければ "なし"}

DESIGN:
{設計案の要約。なければ "なし"}
```

ユーザーの発言を含める場合は verbatim（原文そのまま）で引用する。
要約者（あなた）のバイアスがかからないよう注意する。

### Step 4: 実行（Step 0 で選んだルート）

#### Step 4A: ルートA — エージェント起動

Agent ツールで `second-opinion` エージェントを起動する。
プロンプトには Step 3 で組み立てたコンテキストブロックをそのまま渡す:

```
以下のコンテキストに対して、独立したセカンドオピニオンを提供してください。
あなたはこの会話の履歴を持っていません。渡されたコンテキストのみで判断してください。

{コンテキストブロック}
```

エージェント設定:
- subagent_type は指定しない（agents/second-opinion.md が自動解決される）
- name: "second-opinion"
- model は指定しない（エージェント定義の sonnet を使用）

#### Step 4B: ルートB — codex 外部 cold-read

1. Bash ツールで `codex exec --help` を実行し、そのバージョンにおける非対話実行の
   フラグ（サンドボックス/承認モード/モデル指定/プロンプトの渡し方）を都度確認する。
   フラグはバージョン依存のため本 SKILL.md にハードコードしない。
2. 確認したフラグの中から **読み取り専用相当の最も安全なサンドボックス/承認設定**
   （ファイル書き込み・ネットワークアクセス・コマンド実行の自動承認を伴わないもの）を選ぶ。
3. Step 3 で組み立てたコンテキストブロックのみをプロンプトとして渡す。
   Route A と同じ cold-read 原則を維持するため、**頭（メインセッション）の結論・評価・
   推奨は一切含めない**（渡してよいのは Step 3 のコンテキストブロックのみ）。
4. 非対話実行する（実行例。プレースホルダは Step 4B-1 で確認した実際のフラグ・値に置き換える。
   このブロックは dogfood gate の対象外にするため意図的に言語タグなしで記載している）:

```
codex exec <read-only相当のサンドボックス/承認フラグ> "<Step3のコンテキストブロック>"
```

5. 非ゼロ終了やエラー出力が出た場合は Step 0 のフォールバック方針に従い、
   「codex 不可のためルートA で実行します（理由: ...）」を表示してから Step 4A に切り替える。

### Step 5: 結果の提示

出力を verbatim で提示する（要約・編集をしない）。ラベルは実行したルートに合わせる:

```
SECOND OPINION (Claude Agent, ルートA):
════════════════════════════════════════════════════════════
{エージェント出力をそのまま。要約や編集をしない}
════════════════════════════════════════════════════════════
```

```
SECOND OPINION (codex, ルートB — 外部 cold-read):
════════════════════════════════════════════════════════════
{codex の生出力をそのまま。要約や編集をしない}
════════════════════════════════════════════════════════════
```

提示した後、頭（呼び出し元）がこの生出力を咀嚼して Step 6 の合成に進む
（codex の判断そのものを鵜呑みにせず、頭が解釈・統合する）。

### Step 6: クロスパースペクティブ合成

セカンドオピニオンの出力（ルートA/B いずれも）を提示した後、3-5 箇条書きで合成を提供する:

- **一致点**: 自分（メインセッションの Claude）とセカンドオピニオンが同意する点
- **相違点**: 意見が分かれる点と、その理由
- **前提への影響**: セカンドオピニオンが前提に挑戦した場合、それが推奨にどう影響するか

### Step 7: ユーザーアクション（任意）

セカンドオピニオンが前提に挑戦した場合、ユーザーに確認する:

```
セカンドオピニオンが前提「{前提テキスト}」に挑戦しています。
A) この前提を修正する
B) 元の前提を維持する
```

ユーザーが理由を添えて B を選んだ場合、それは重要なシグナルとして記録する。

## エラーハンドリング

- エージェント起動に失敗した場合（ルートA）: エラーを報告し、手動でのレビューを提案する
- エージェントが空の応答を返した場合（ルートA）: 再試行を 1 回だけ行い、それでもダメなら報告する
- タイムアウト（ルートA）: Agent ツールのデフォルトタイムアウトに従う
- `codex exec` が非ゼロ終了・エラー出力を返した場合（ルートB）: 握りつぶさず
  「codex 不可のためルートA で実行します」と表示してルートAへフォールバックする
- codex CLI が未インストール・未認証の場合: Step 0 の `command -v codex` で検出できず
  そもそもルートBに入らない（ルートAのみで動作）

## 使用例

```
# office-hours 後にセカンドオピニオン
/evolve-anything:second-opinion startup

# 設計レビュー
/evolve-anything:second-opinion general

# codex 外部 cold-read を明示指定（codex 検出時のみ有効）
/evolve-anything:second-opinion general "マイクロサービス vs モノリスの設計判断" codexで

# モード自動判定（会話文脈から）
/evolve-anything:second-opinion

# コンテキスト直接指定
/evolve-anything:second-opinion general "マイクロサービス vs モノリスの設計判断"
```
