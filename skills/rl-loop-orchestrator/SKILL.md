## Arguments
- `--target PATH`: 改善対象のスキルファイルパス（必須）
- `--loops N`: ループ回数（デフォルト: 1）
- `--auto`: 人間確認ステップをスキップ（デフォルト: false、要確認）
- `--dry-run`: 構造テスト（実際の変更は行わない）
- `--output-dir DIR`: 出力ディレクトリ（デフォルト: `.rl-loop/`）

## 概要

3つの役割が連携して1ループを回す:

```
[実装者] スキル/ルールの変更を適用
    ↓
[採点者] 技術品質 + ドメイン品質 + 構造品質で統合スコアを算出
    ↓
[進化者] genetic-prompt-optimizer で次世代を生成
    ↓
[人間] スコアと変更差分を確認 → 承認/却下
    ↓
(次ループへ)
```

## ワークフロー

### Step 1: 現状ベースライン取得

1. 対象スキルの現在の内容を読み込む
2. rl-scorer エージェントを起動してベースラインスコアを取得
3. スコアと現状を出力ディレクトリに記録

### Step 2: バリエーション生成

1. `/optimize --target PATH --generations 1 --population 3 --dry-run` で構造確認
2. genetic-prompt-optimizer で3つのバリエーションを生成
3. 各バリエーションの差分を表示

### Step 3: 評価

1. 各バリエーションを仮適用
2. rl-scorer エージェントで各バリエーションを採点
3. ベースラインと比較

### Step 4: 選択と人間確認

1. 最高スコアのバリエーションを提示
2. ベースラインとの差分を表示
3. **人間が承認/却下を判断**（`--auto` でスキップ可能）
4. 承認された場合のみ対象スキルに反映

### Step 5: 記録

1. ループ結果を出力ディレクトリに記録
2. スコア推移グラフ用のデータを蓄積

## 採点エージェント

rl-scorer エージェント（`agents/rl-scorer.md`）が統合スコアを算出する。

スコア構成:
- **技術品質スコア** (40%): 明確性・完全性・一貫性・エッジケース・テスト可能性
- **ドメイン品質スコア** (40%): プロジェクトの CLAUDE.md からドメインを推定し、適切な評価軸を自動選択
- **構造スコア** (20%): スキルファイルの構造的品質

## 出力先

```
.rl-loop/                         # デフォルト（--output-dir で変更可）
├── {run_id}/
│   ├── baseline.json      # ベースラインスコア
│   ├── variants/           # 各バリエーション
│   │   ├── variant_0.md
│   │   ├── variant_1.md
│   │   └── variant_2.md
│   ├── scores.json         # 各バリエーションのスコア
│   └── result.json         # 最終結果
└── history.jsonl           # 全ループの履歴
```

## 依存スキル/エージェント

| 依存先 | 用途 |
|--------|------|
| genetic-prompt-optimizer | バリエーション生成 |
| rl-scorer エージェント | 統合スコア算出 |

## 注意事項

- 最初のループは必ず `--dry-run` なしで人間確認ステップを含める
- スコアが下降した場合は自動的にバックアップから復元
- 1ループの API コスト目安: sonnet × 3回（バリエーション生成）+ haiku × 3回（評価）
