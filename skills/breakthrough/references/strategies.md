## Type A: 評価関数を先に定義する

### 症状
「完成がどんな状態か言語化できていない」「なんとなくいい感じになってほしい」

### なぜ行き詰まるか
評価基準が曖昧だとエージェントは毎回違うものを最適化してしまう。何回反復しても「違う」としか言えない状態になる。

### 手順
1. ユーザーと一緒に「完成の定義」を作る
2. 評価基準を具体的なチェックリストに落とす
3. 各基準に0〜10点のスコアを割り当て、合格ラインを決める
4. そのスコアシートをエージェントへの指示に含める

### 評価関数テンプレート
```
## 完成の定義（スコアシート）

| 基準 | 配点 | 評価方法 |
|------|------|---------|
| [基準1] | /10 | [どうやって判定するか] |
| [基準2] | /10 | [どうやって判定するか] |
| [基準3] | /10 | [どうやって判定するか] |

合格ライン: XX点 / 30点以上
```

### ポイント
評価基準は「人間が見て判定できるもの」にする。「いい感じ」「自然」は基準にならない。「色の差分がΔE<5」「フォントサイズ誤差<2px」のように測定可能にする。

## Type B: Tutor-Student 非対称ロール

### 症状
「複数エージェントで並列に試したが全員同じ結果」「いくら試しても同じ壁にぶつかる」

### なぜ行き詰まるか
並列エージェントは同じモデル・同じコンテキスト・同じプロンプトで動くので、同じ盲点を全員が共有している。並列数を増やしても盲点は崩せない。

### 手順
1. 現在の「実装者エージェント」は Student (Coder) として続投
2. 新たに Tutor (Reviewer) エージェントを追加する
3. Tutor の役割: Student の出力を見て「何が足りないか・何が違うか」だけを指摘する。**解答は絶対に言わない**
4. Student は Tutor のフィードバックだけを受け取って改善する
5. 最大 N ループ（デフォルト: 5回）で終了

### 非対称性が重要な理由
Tutor が解答を言ってしまうと、Student はただのコピー機になる。Tutor は「なぜ違うか」の視点を与えるだけで、実際の解は Student が生成する。この非対称性が盲点突破のカギ。

### エージェントプロンプトの骨子
→ `agent-templates.md` の「Tutor-Student テンプレート」を参照

## Type C: Solver → Reviewer → Refiner 反復ループ（MAgICoRe）

### 症状
「改善はしているが正解に近づいているか分からない」「ループしているが差分が縮まらない」

### なぜ行き詰まるか
生成と評価を同じエージェントがやっていると、自分の出力を自分で評価することになり、既存の思い込みが修正されない。役割を分離することで客観的な評価が入る。

### 手順
1. **Solver**: 問題を解く。初期解または前回 Refiner の出力を受け取って改善案を生成
2. **Reviewer**: Solver の出力を評価。改善点を **スコア付き箇条書き** で出力。解答は出さない
3. **Refiner**: Reviewer のフィードバックだけを受け取って Solver の出力を改善
4. Reviewer スコアが合格ラインを超えるか、最大5回で終了

### スコアリングの書き方（Reviewerへの指示）
```
評価は以下の形式で出力せよ:

総合スコア: X / 10

改善点（優先度順）:
1. [最も重要な問題] - 重要度: 高
2. [次の問題] - 重要度: 中
3. ...

合格判定: [合格/不合格]
理由: [1行]
```

### ポイント
Reviewer には評価基準（Type A で作ったスコアシート）を必ず渡す。基準なしのレビューは「なんとなく悪い」という曖昧なフィードバックになって収束しない。

## Type D: インプット強化

### 症状
「AIが推測で動いていて実際のデータに合っていない」「コンテキストが足りない気がする」

### なぜ行き詰まるか
エージェントは与えられた情報だけで動く。足りない情報は推測で補うが、その推測が間違っていると何度改善しても正解に近づかない。

### 手順
1. エージェントが「推測していること」を列挙する（プロンプトに「あなたが仮定していることを全部書いてください」と聞く）
2. 推測を事実で置き換えられるデータ・ファイル・情報を特定する
3. 投入方法を選ぶ:
   - ファイルを直接渡す（Read ツール / ファイルパス指定）
   - スクリーンショットを撮って渡す（UI再現問題）
   - MCP サーバーで直接データを取得させる（Figma MCP、DB MCP など）
   - API レスポンスをそのままプロンプトに含める

### Figma 再現問題の場合
Figma MCP が利用可能な場合: デザイントークン・Auto Layout・コンポーネント名を直接取得させる。AIの推測（「たぶんこの色」）を実データで置き換えることで再現度が大幅に向上する。

## Type E: 視点転換（Perspective Shift）

### 症状
「同じアプローチを繰り返しているが突破できない」「アプローチを変えたいが何を変えればいいか分からない」「なんか行き詰まってる感じがする（タイプが特定できない）」

### なぜ行き詰まるか
問題への最初のアプローチが「正しい方向」という前提が固定化してしまっている。別の角度から見れば自明な解があっても、既存のフレームの中では見えない。

### 手順
以下の視点転換の問いを使い、新しいアプローチを発見する:

#### 1. 逆転アプローチ
「完成した状態から逆算すると何が必要か？」
→ 目標から逆算することで「何を最初に作るべきか」が変わることがある

#### 2. 制約除去
「今の制約（言語・ツール・アーキテクチャ）がなければどう解くか？」
→ 制約を外して考えた解を見た後で、現実的な制約の中で近似する

#### 3. 異分野応用
「別の分野ではこの問題をどう解いているか？」
→ UI 再現問題なら「印刷業のカラーキャリブレーション」アプローチ
→ コード最適化なら「コンパイラの最適化パス」アプローチ
→ 文章生成なら「文体模写・翻案」アプローチ

#### 4. ゼロベース再設計
「今まで作ったものを全部捨てて、最小限から作り直したら何が残るか？」
→ 蓄積された複雑さが突破を妨げていることがある

#### 5. 問題の再定義
「そもそも解こうとしている問題は本当にその問題か？」
→ 「Figmaに100%近づける」ではなく「ユーザーが満足する UI を作る」ならアプローチが変わる

### エージェントへの渡し方
```
あなたは Devil's Advocate（悪魔の代弁者）として動いてください。
現在のアプローチ: [現状説明]

以下の問いに答えてください:
1. このアプローチが根本的に間違っている場合、何が原因か？
2. まったく別の方向から解くとしたら何を試すか（3案）？
3. 別の分野ではこの問題をどう解いているか？

答えは批判的で構いません。現在のアプローチを守る必要はありません。
```

### ポイント
Type E は「どのタイプか判断できない」「全部試した」という時の最終手段でもある。視点転換エージェントには既存のコンテキスト（これまで試したこと）を全部渡すのが効果的。

## 複数タイプの組み合わせパターン

| パターン | 推奨組み合わせ |
|---------|--------------|
| 定義が曖昧 + 並列で詰まっている | A → B（まず定義を作ってからTutor-Student） |
| ループはしているが評価できない | A → C（スコアシートを作ってからMAgICoRe） |
| 全部試した感がある | E → B（視点転換で方向を変えてからTutor-Student） |
| データが足りない + 方向不定 | D → C（インプット強化してからMAgICoRe） |
