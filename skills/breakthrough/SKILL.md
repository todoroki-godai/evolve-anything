# Breakthrough — 行き詰まり突破スキル

複数のエージェントに同じ問題を並列で見させても突破できないのは、**全員が同じ盲点を共有しているから**。
このスキルは「なぜ惜しい状態で止まっているか」を診断し、盲点を崩す戦略を提案・実行する。

---

## Phase 1: Intake — 状況を整理する

ユーザーの説明（またはコンテキスト）から以下を抽出する。不明な項目は最初に一括確認する。

| 項目 | 内容 | 空白時の対処 |
|------|------|------------|
| 問題の本質 | 何を達成しようとしているか | 必ず確認 |
| 現在の到達点 | 「惜しい」状態の具体描写 | 必ず確認 |
| 試したこと | これまでのアプローチ一覧 | 確認推奨 |
| 「完成」の定義 | 何をもって成功とするか | **不明なら必ず確認。これがないと突破できない** |

> 「完成の定義」が曖昧なケースは多い。「なんとなくいい感じ」は評価関数として機能しない。
> 数値・チェックリスト・比較対象など、具体的な形に落とすことが突破の第一歩になる。

---

## Phase 1.5: 仮説ツリー初期化（VeriTrace Phase 1）

Phase 1 で状況を整理したら、**調査を開始する前に仮説を列挙する**。
仮説を可視化することで「前半の仮説を忘れる」「調査方向が気づかずに変わる」問題を防ぐ。

### 仮説リストの形式

```
## 現在の仮説リスト

| ID | 仮説 | 確信度 | ステータス |
|----|------|--------|----------|
| h1 | [問題の主原因に関する仮説] | 0.5 | active |
| h2 | [別の原因仮説] | 0.3 | active |
| h3 | [前提条件に関する仮説] | 0.7 | active |
```

### 調査中の更新ルール
- 新しい発見を得るたびに各仮説の確信度を更新する
  - 支持証拠: confidence +0.1（最大 1.0）
  - 反証: confidence -0.15（最低 0.0）
- confidence が 0.3 未満になった仮説は `suspended` に変更し、その反証内容を記録する
- `hypothesis_tracker.py`（`scripts/lib/hypothesis_tracker.py`）を使って JSONL に永続化することもできる
  - セッションファイル: `~/.claude/rl-anything/hypothesis_{session_id}.jsonl`
  - `detect_contradiction()` で evidence_against 3件以上の active 仮説ペアを検出できる

### 調査完了時
- 全仮説の最終 status を `confirmed` / `refuted` / `suspended` に更新する
- Phase 5 の最終レポートに仮説の変遷を含める

## Phase 2: Diagnosis — 行き詰まりタイプを判定する

以下の4タイプから**主要タイプを1つ**選ぶ（複数混在している場合は最も根本的なものを選択）。

| タイプ | 症状 | 判定のヒント |
|--------|------|------------|
| **A: 評価曖昧** | 「何が正解か自分でも分からない」 | 完成定義が言語化されていない |
| **B: 同質盲点** | 「複数エージェントで試したが全員同じ結果になる」 | 並列エージェントが収束しない |
| **C: 方向不定** | 「改善しているが正解に近づかない、あるいは近づいているか分からない」 | 反復しても差分が縮まらない |
| **D: 情報不足** | 「AIが推測で動いていて、実際のデータに合っていない」 | コンテキスト・データが不完全 |
| **E: 視点固着** | 「同じアプローチを繰り返しているが突破できない」「どのタイプか分からない」 | 問題へのアプローチ自体が固定化している |

---

## Phase 3: Strategy Selection — 突破戦略を選ぶ

タイプに対応した戦略の詳細は `references/strategies.md` を参照。以下は要点。

### Type A → 評価関数を先に定義する
「完成」を数値とチェックリストで定義し直す。エージェントに渡す前の準備として扱う。
→ 評価関数テンプレートを一緒に作成する。

### Type B → Tutor-Student 非対称ロール
Reviewer 専用エージェントを追加する。Reviewer は **解答を言わず、何が足りないかだけを指摘する**。
これで並列エージェントが共有していた盲点を外から指摘できる。

### Type C → Solver → Reviewer → Refiner 反復ループ（MAgICoRe）
3つの役割を明確に分離し、最大N回のループで収束させる。
Reviewer は改善すべき点をスコア付きで出力し、Refiner はそのフィードバックだけを受け取って改善する。

### Type D → インプット強化
AIが推測で動いている原因を特定し、具体的なデータ・スクリーンショット・ファイル・MCP接続を投入する方法を提案する。

### Type E → 視点転換（Devil's Advocate）
「全部試した」「どのタイプか分からない」「同じアプローチの繰り返しを疑っている」ケース。
問題への最初のアプローチが「正しい方向」という前提を壊す Devil's Advocate エージェントを使う。
逆算・制約除去・異分野応用・問題の再定義の4つの問いで新しい方向性を見つける。

---

## Phase 4: Proposal — 戦略を提示して承認を得る

以下の形式で提案を出す。`AskUserQuestion` ツールで GO/NO を確認する。

```
## 診断結果
タイプ: [A/B/C/D] - [タイプ名]

根拠: [なぜこのタイプと判定したか 1〜2行]

## 突破戦略
[戦略名と概要]

## エージェント構成
[役割と責任の明示。テキスト図で表現]

## 終了条件
[スコア閾値 or 最大ループ回数 or 合格基準]

## 次の一手
[ユーザーがまず何をすべきか、1アクション]
```

---

## Phase 5: Execution — エージェントを起動する

承認後、選択された戦略に応じて `Agent` または `Task` ツールでエージェントを起動する。

各エージェントのプロンプトには必ず以下を含める:
1. **役割と責任の明示**（Solver/Reviewer/Refiner/Tutor/Student のいずれか）
2. **他の役割との違い**（例: 「あなたは解答を出してはいけない。批評だけが役割」）
3. **終了条件**（何が揃ったらループを終えるか）
4. **出力フォーマット**（次のエージェントが受け取れる形式）

エージェントプロンプトの具体例は `references/agent-templates.md` を参照。

---

## 重要な原則

- **評価関数のない反復は収束しない**。Phase 1 で「完成の定義」を得るまで先に進まない。
- **並列は収束を速めるが、盲点は崩せない**。盲点を崩すには非対称ロールが必要。
- **ループ回数に上限を設けること**。無制限ループはトークン浪費になる。デフォルト上限は5回。
- **詳細戦略は references/ を読む**。SKILL.md は判断のためのガイドで、実行手順の詳細は参照ファイルにある。
